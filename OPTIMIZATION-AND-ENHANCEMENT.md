# SNH48 Live Enhancer — 优化与功能增强建议

> 基于项目 v2.2.0 代码库的全面分析，聚焦代码质量改进与功能增强，不涉及分发与部署层面。

---

## 一、代码质量改进

### 1.1 代码可读性与可维护性

#### 1.1.1 content.js 单文件体量过大（约 2260 行）

**问题**：content.js 承载了暗黑模式、搜索、索引、视频控制、导航、提醒、浮动面板等全部功能，是典型的"上帝对象"反模式。函数之间通过闭包变量隐式耦合，修改任一功能可能产生不可预见的副作用。

**建议**：
- 按功能域拆分为独立模块文件（如 `dark-mode.js`、`search.js`、`video-enhancer.js`、`member-index.js`、`float-panel.js`、`reminder.js`、`quick-nav.js`），通过共享的 `config.js` 和 `utils.js` 传递配置与工具函数
- 使用 ES Module（`import`/`export`）或 IIFE 命名空间模式组织模块边界
- 在 manifest.json 的 `content_scripts` 中按序引入各模块

**优先级**：高 | **实施难度**：中

#### 1.1.2 background.js 同样存在职责过重问题（约 1180 行）

**问题**：background.js 同时负责索引存储管理、HTML 解析、批量抓取调度、搜索加速索引、消息路由、上下文菜单等，修改搜索逻辑可能影响索引写入，修改抓取策略可能影响消息通信。

**建议**：
- 拆分为 `index-store.js`（存储与搜索）、`index-crawler.js`（批量抓取与解析）、`message-handler.js`（消息路由）、`html-parser.js`（HTML 解析）
- 各模块通过明确的接口交互，而非共享全局状态

**优先级**：高 | **实施难度**：中

#### 1.1.3 HTML 字符串拼接构建 UI

**问题**：浮动面板（content.js 第 1718-1816 行）和搜索结果面板均通过字符串拼接构建 DOM，可读性差、难以维护、XSS 风险高（虽有 `escapeHtml`，但依赖开发者每次都正确使用）。

**建议**：
- 使用 `document.createElement` + 属性设置的方式构建 DOM，或引入轻量级模板辅助函数
- 将面板 HTML 模板提取为独立文件，通过 `fetch(chrome.runtime.getURL(...))` 加载后填充数据

**优先级**：中 | **实施难度**：低

### 1.2 编码模式不一致

#### 1.2.1 JavaScript 语言风格混用

**问题**：
- content.js 使用 ES5 语法（`var`、`function(){}`、`Array.prototype.forEach`）
- background.js 使用 ES6+ 语法（`const`/`let`、箭头函数、`async/await`、模板字符串、解构赋值）
- popup.js 使用 ES5 语法但混用了 `log(...args)` 剩余参数（ES6）

**建议**：
- 统一采用 ES6+ 语法（Chrome MV3 扩展的最低运行环境 Chrome 89 完全支持 ES6+）
- 制定并遵循统一的代码风格指南，包括变量声明方式、函数定义方式、字符串引号等
- 考虑引入 ESLint 配置文件强制执行

**优先级**：中 | **实施难度**：低

#### 1.2.2 DEFAULT_CONFIG 三处重复定义

**问题**：`DEFAULT_CONFIG` 在 content.js（第 139 行）、background.js（第 28 行）、popup.js（第 12 行）各定义了一次，字段和默认值必须手动保持同步，极易产生不一致。

**建议**：
- 提取为独立的 `config.js` 模块，三处统一引用
- 或在 background.js 中作为唯一权威来源，content.js 和 popup.js 通过消息机制获取

**优先级**：高 | **实施难度**：低

#### 1.2.3 日志函数实现不一致

**问题**：
- content.js 中 `log()` 为空函数体（生产禁用），`warn()` 和 `error()` 保留功能
- background.js 中 `log()` 同样为空函数体
- popup.js 中 `log()` 使用 ES6 剩余参数 `...args`，但其余代码为 ES5 风格
- 三处日志前缀不同但逻辑重复

**建议**：
- 统一日志工具模块，支持级别控制（debug/info/warn/error）
- 通过配置或环境变量控制日志级别，而非注释掉函数体
- 开发环境启用 debug 级别，生产环境仅保留 warn 和 error

**优先级**：低 | **实施难度**：低

---

### 1.3 冗余与重复代码

#### 1.3.1 暗黑模式颜色检测函数高度相似

**问题**：`_hasBrightBg()`、`_hasDarkText()`、`_hasDimText()` 三个函数（content.js 第 280-378 行）结构几乎相同：均解析 style 字符串 → 提取颜色值 → 判断阈值，仅阈值参数不同。hex 解析和 rgb 解析逻辑在三个函数中各重复一次。

**建议**：
- 提取通用的颜色解析函数 `parseColorFromStyle(styleText, property)`，返回 `{r, g, b}` 或 `null`
- 提取颜色分类函数 `classifyColor(r, g, b)`，返回 `"bright"` / `"dark"` / `"dim"` / `"normal"`
- 三个原函数简化为对上述工具函数的组合调用

**优先级**：中 | **实施难度**：低

#### 1.3.2 Extension Context 失效处理代码重复

**问题**：`safeSendMessage()` 和 `safeStorageSet()` 中的失效检测和清理逻辑（content.js 第 89-136 行）几乎完全相同：检测 `context invalidated` → 设置 `_extensionDead = true` → 清理所有定时器。

**建议**：
- 提取通用的 `handleContextInvalidated()` 函数
- `safeSendMessage` 和 `safeStorageSet` 在检测到失效时统一调用

**优先级**：低 | **实施难度**：低

#### 1.3.3 搜索结果项的 scroll-into-view 逻辑重复

**问题**：`collectAllSearchable()` 中多个搜索来源（即将开始、参演成员、成员人气榜）的 action 回调均包含相同的 `scrollIntoView + outline 高亮 + setTimeout 清除` 模式。

**建议**：
- 提取 `scrollToAndHighlight(element, duration)` 工具函数
- 各 action 回调统一调用此函数

**优先级**：低 | **实施难度**：低

### 1.4 错误处理机制不足

#### 1.4.1 批量索引缺乏全局错误恢复

**问题**：background.js 的 `runRangeIndex()` 中，如果 Service Worker 在索引过程中被 Chrome 回收（MV3 的 30 秒空闲限制），正在进行的索引进度会丢失。虽然 `indexingState` 保存了 `currentId`，但 Service Worker 重启后状态不会自动恢复。

**建议**：
- 将 `indexingState` 持久化到 `chrome.storage.session`（MV3 专用的会话级存储，Service Worker 重启后仍可用）
- Service Worker 启动时检查是否有未完成的索引任务，自动恢复执行
- 或使用 `chrome.alarms` API 定期唤醒 Service Worker 继续索引

**优先级**：高 | **实施难度**：中

#### 1.4.2 搜索结果合并缺乏冲突处理

**问题**：content.js 的 `performSearch()` 中，本地搜索结果与后台搜索结果通过 URL 去重后合并，但如果同一 URL 在本地和后台有不同的标题或成员信息（因数据更新），没有合并策略，仅保留先出现的数据。

**建议**：
- 定义明确的合并策略：后台索引数据优先（因为批量索引时解析更完整），或按 `indexedAt` 时间戳取最新
- 合并时保留两个来源的元信息，避免数据丢失

**优先级**：中 | **实施难度**：低

#### 1.4.3 截图功能缺乏跨域视频检测

**问题**：`takeScreenshot()` 使用 canvas 的 `drawImage` 绘制视频帧，但如果视频源是跨域的（CORS 限制），canvas 会被"污染"，`toBlob()` 将抛出 SecurityError。当前仅用 try-catch 捕获，但未给用户明确的错误提示。

**建议**：
- 捕获 SecurityError 时，显示"当前视频源不支持截图（跨域限制）"等明确提示
- 尝试在视频元素上设置 `crossOrigin="anonymous"` 属性（需服务端配合）

**优先级**：低 | **实施难度**：低

#### 1.4.4 HTML 解析容错性不足

**问题**：background.js 的 `parseHtmlSimple()` 使用正则表达式解析 HTML，对格式变化极为脆弱。例如标题提取依赖 `<span class="title1">` 的精确匹配，如果站点改版增加属性或改变标签嵌套，解析将静默失败。

**建议**：
- 在解析失败时记录详细的诊断信息（原始 HTML 片段、匹配尝试结果），便于排查
- 增加解析结果的有效性校验：标题为空时标记为 `parse_failed` 而非返回空对象
- 考虑使用 `DOMParser` 在 Service Worker 的 OffscreenCanvas 环境中解析（MV3 支持 Offscreen API）

**优先级**：中 | **实施难度**：中

### 1.5 注释与文档不足

#### 1.5.1 消息协议缺乏文档

**问题**：content.js 与 background.js 之间通过 `chrome.runtime.sendMessage` 通信，消息类型包括 `INDEX_CURRENT_PAGE`、`SEARCH_INDEX`、`START_RANGE_INDEX`、`STOP_RANGE_INDEX`、`GET_INDEXING_STATE`、`GET_INDEX_STATS`、`CLEAR_FULL_INDEX`、`CONFIG_UPDATED`、`GET_CONFIG`、`INDEXING_PROGRESS`、`INDEXING_COMPLETE`、`SHOW_INDEXING_STATUS` 等，但缺乏统一的消息协议文档。

**建议**：
- 在代码中或独立文件中维护消息类型注册表，包含类型名、发送方、接收方、请求/响应格式
- 使用常量对象定义消息类型，避免字符串硬编码散落各处

**优先级**：中 | **实施难度**：低

#### 1.5.2 关键算法缺乏说明

**问题**：
- 暗黑模式的颜色阈值（如 RGB >= 220 判定为亮色、RGB <= 120 判定为深色）缺乏设计依据说明
- LRU 裁剪的 80% 保留策略缺乏说明
- 搜索加速索引的 2-gram/3-gram 分词策略缺乏说明
- 批量索引的并发数 3、延迟 200ms/500ms 等参数缺乏调优依据

**建议**：
- 在代码中添加设计决策注释（为何选择这些阈值、参数）
- 或在项目文档中记录调优过程和基准测试结果

**优先级**：低 | **实施难度**：低

### 1.6 性能瓶颈

#### 1.6.1 MutationObserver 全量扫描

**问题**：暗黑模式的 MutationObserver（content.js 第 437-463 行）监听 `document.documentElement` 的 `childList + subtree + attributes(style)` 变更。每次 DOM 变动都触发 `_cleanInlineStyles()`，对新增节点执行全量颜色解析。在动态内容频繁的页面（如直播聊天、弹幕区域），可能导致大量不必要的计算。

**建议**：
- 对 MutationObserver 回调增加防抖（debounce），合并短时间内的多次变更
- 缓存已处理过的元素（通过 WeakSet），避免重复处理
- 缩小观察范围：仅观察主要内容区域（如 `.content`），而非整个 `documentElement`
- 对 `attributes` 变更，仅当 style 属性值实际改变时才重新处理

**优先级**：高 | **实施难度**：中

#### 1.6.2 搜索结果 DOM 频繁重建

**问题**：`performSearch()` 中，每次搜索结果更新都完全清空并重建 `resultsEl.innerHTML`（content.js 第 755 行），包括过滤 chips、分组、结果项。在快速输入时（虽有 150ms 防抖），仍可能导致明显的界面闪烁。

**建议**：
- 实现 DOM diff 机制：比较新旧数据，仅更新变化的部分
- 或使用虚拟列表（virtual list）模式，仅渲染可见区域的结果项
- 过滤 chips 在数据源不变时缓存，不随搜索结果重建

**优先级**：中 | **实施难度**：中

#### 1.6.3 归档分片搜索效率低

**问题**：background.js 的 `loadArchivedShards()`（第 490-504 行）在搜索无结果时串行加载所有归档分片，每个分片都是一次 `chrome.storage.local.get` 异步调用。如果有 10 个分片，需要 10 次串行 I/O。

**建议**：
- 一次性批量读取所有分片键（`chrome.storage.local.get(shardKeys)`），减少 I/O 次数
- 为归档数据也建立轻量级索引（如成员名→分片号的映射），实现按需加载
- 考虑在索引建立时预生成搜索摘要，避免搜索时全量扫描

**优先级**：中 | **实施难度**：中

#### 1.6.4 Shadow DOM 遍历性能问题

**问题**：视频快捷键功能中查找 video 元素时（content.js 第 1371-1378 行），遍历 `document.querySelectorAll("*")` 检查每个元素的 `shadowRoot`，在 DOM 节点较多的页面上性能开销大。

**建议**：
- 缓存 video 元素引用，仅在 DOM 变更时重新查找
- 使用 `document.querySelector("video")` 优先，仅在未找到时才遍历 Shadow DOM
- 或使用 MutationObserver 监听 video 元素的插入

**优先级**：中 | **实施难度**：低

#### 1.6.5 成员索引保存策略可优化

**问题**：`saveMemberIndex()` 使用 800ms 节流，但在页面卸载时依赖 `pagehide`/`beforeunload` 事件同步保存。如果索引数据较大（接近 4MB 上限），`JSON.stringify` 可能阻塞主线程。

**建议**：
- 对 `JSON.stringify` 增加耗时监控，超过阈值时警告
- 考虑增量序列化（仅序列化变更部分）或使用 `structuredClone`
- 在 `pagehide` 中使用 `navigator.sendBeacon` 或设置 `event.persisted` 处理

**优先级**：低 | **实施难度**：中

## 二、功能增强建议

### 2.1 缺失的核心功能

#### 2.1.1 配置导入/导出

**问题**：用户更换浏览器或重装扩展后，所有配置和索引数据丢失，需要重新建立。

**建议**：
- 在 popup 或浮动面板中增加"导出配置"和"导入配置"按钮
- 导出为 JSON 文件，包含用户配置和成员索引数据
- 导入时校验数据格式，提示是否覆盖现有数据

**优先级**：高 | **实施难度**：低

#### 2.1.2 播放历史记录

**问题**：当前无任何观看历史功能，用户无法快速找到之前看过的公演回放。

**建议**：
- 自动记录用户访问的公演页面（URL、标题、访问时间、观看进度）
- 在搜索结果中标记已观看的公演
- 增加"最近观看"快捷入口
- 数据存储于 `chrome.storage.local`，与成员索引类似采用 LRU 淘汰

**优先级**：中 | **实施难度**：中

#### 2.1.3 视频播放进度记忆

**问题**：用户刷新页面或重新打开回放时，视频从头开始播放，无法自动跳转到上次观看位置。

**建议**：
- 定期（如每 10 秒）记录当前视频的 `currentTime` 到 `chrome.storage.local`
- 页面加载时检测是否有该 URL 的进度记录，自动跳转
- 提供清除单条或全部进度记录的选项

**优先级**：高 | **实施难度**：低

#### 2.1.4 公演日历视图

**问题**：当前仅通过列表形式展示即将开始的公演，无法直观查看某段时间内的公演安排。

**建议**：
- 基于已索引的公演数据（含日期信息），提供日历视图
- 标记有公演的日期，点击可查看当日公演列表
- 支持按分团筛选
- 可作为浮动面板的一个标签页或独立弹窗

**优先级**：中 | **实施难度**：中

---

### 2.2 用户体验改进

#### 2.2.1 浮动面板位置记忆改进

**问题**：面板位置存储在 `localStorage`（content.js 第 1835 行），但 `localStorage` 在 `live.48.cn` 域下，不同标签页共享。如果用户在多个标签页打开网站，面板位置可能冲突。且 `localStorage` 不受扩展的 `storage` 权限管理，清除浏览器数据时会被误删。

**建议**：
- 改用 `chrome.storage.sync` 存储面板位置，跨设备同步
- 或使用 `chrome.storage.local`，与扩展其他数据统一管理

**优先级**：低 | **实施难度**：低

#### 2.2.2 搜索结果键盘导航体验

**问题**：搜索结果的键盘导航（content.js 第 1054-1081 行）使用 `currentIndex` 在扁平化列表中移动，但展开分组后索引可能错位（因为 `visibleItems` 的 DOM 顺序与 `flatItems` 的逻辑顺序可能不一致）。

**建议**：
- 确保 `flatten()` 的结果与 DOM 渲染顺序严格一致
- 在分组展开/折叠时重新计算 `flatItems`
- 增加 Home/End 键支持（跳到首/末项）

**优先级**：中 | **实施难度**：低

#### 2.2.3 Toast 通知队列

**问题**：`showToast()` 复用同一个 `.snh48-toast` 元素，快速连续触发时新消息会覆盖旧消息，用户可能错过重要提示。

**建议**：
- 实现 Toast 队列：新消息入队，当前消息消失后显示下一条
- 或支持多条 Toast 同时显示（垂直堆叠，自动消失）

**优先级**：低 | **实施难度**：低

#### 2.2.4 快捷键自定义

**问题**：视频快捷键硬编码，用户无法自定义按键绑定。部分快捷键（如 `S` 截图、`[`/`]` 调速）可能与用户习惯冲突。

**建议**：
- 在浮动面板中增加快捷键自定义界面
- 配置存储于 `chrome.storage.sync`
- 支持快捷键冲突检测和提示

**优先级**：中 | **实施难度**：中

#### 2.2.5 搜索结果持久化与预加载

**问题**：每次打开搜索框都需要重新执行搜索，即使查询词未变。在索引数据量大时，首次搜索可能有明显延迟。

**建议**：
- 缓存最近的搜索结果（LRU 缓存，最多 20 条查询）
- 在搜索框获得焦点时预加载热门/最近搜索结果
- 搜索结果中增加加载状态指示器

**优先级**：低 | **实施难度**：低

---

### 2.3 集成与数据管理

#### 2.3.1 索引数据版本管理

**问题**：当 HTML 解析逻辑更新后（如 background.js 的 `parseHtmlSimple` 修复了某个解析 bug），已存储的旧索引数据不会自动更新，导致搜索结果中存在过时或错误的数据。

**建议**：
- 为索引数据增加版本号（schema version）
- 解析逻辑更新时递增版本号
- 检测到版本不匹配时，自动触发增量重新索引（仅重新解析已有 URL）
- 或在搜索结果中标记数据版本，提示用户可能需要更新

**优先级**：高 | **实施难度**：中

#### 2.3.2 增量索引（而非全量重建）

**问题**：批量索引（`startRangeIndex`）在开始时会清空所有已有索引数据（background.js 第 939-946 行），即使大部分数据仍然有效。用户如果想补充索引新的 ID 范围，必须重新索引所有数据。

**建议**：
- 支持增量索引模式：仅索引尚未索引过的 ID
- 保留已有数据，新数据合并入库
- 提供"全量重建"和"增量补充"两种模式供用户选择

**优先级**：高 | **实施难度**：中

#### 2.3.3 索引数据压缩

**问题**：索引数据以 JSON 明文存储在 `chrome.storage.local`，大量重复的字符串（如 URL 前缀 `https://live.48.cn/Index/invideo/club/`、成员名、公演标题中的重复词）占用不必要的空间。

**建议**：
- 对 URL 使用相对路径存储（仅保留 `/club/X/id/Y` 部分），查询时拼接前缀
- 对成员名使用字符串表（string table）去重
- 对公演标题中的常见后缀（"剧场公演"、"特别公演"）使用缩写
- 评估使用 `CompressionStream` API 压缩存储的可行性

**优先级**：中 | **实施难度**：中

#### 2.3.4 多标签页协调

**问题**：当用户在多个标签页打开 `live.48.cn` 时，每个标签页都会独立运行 content.js，可能导致：
- 多个标签页同时向 background.js 发送 `INDEX_CURRENT_PAGE` 消息
- 多个浮动面板同时存在
- 配置变更时需要逐个标签页同步

**建议**：
- 使用 `chrome.storage.onChanged` 监听配置变更，替代当前的消息通知机制
- 仅在活跃标签页显示浮动面板和搜索入口
- 索引操作通过 background.js 统一调度，避免重复索引

**优先级**：中 | **实施难度**：中

---

### 2.4 安全与验证

#### 2.4.1 搜索输入净化

**问题**：搜索输入框的值直接传入 `performSearch()` 和 `collectAllSearchable()`，虽然 `escapeHtml()` 在渲染时处理了 XSS，但搜索逻辑中的 `String.prototype.includes()` 和正则匹配未对特殊字符做处理。恶意输入（如超长字符串、正则特殊字符）可能导致性能问题。

**建议**：
- 限制搜索输入的最大长度（如 100 字符）
- 过滤或转义正则特殊字符
- 对搜索频率增加限流（当前仅 150ms 防抖，无硬性限制）

**优先级**：中 | **实施难度**：低

#### 2.4.2 批量索引的请求频率控制

**问题**：批量索引以 3 并发 + 200ms 延迟的速率请求 `live.48.cn`，虽然已做了自适应延迟（出错时 500ms），但没有全局速率限制。如果用户频繁启动大范围索引，可能对目标站点造成压力或触发反爬。

**建议**：
- 增加全局请求速率限制（如每秒最多 5 个请求）
- 检测到 429（Too Many Requests）响应时自动退避
- 在 UI 中显示当前请求速率，让用户了解影响

**优先级**：中 | **实施难度**：低

#### 2.4.3 存储数据完整性校验

**问题**：索引数据存储在 `chrome.storage.local`，但没有完整性校验。如果存储数据损坏（如浏览器崩溃导致写入不完整），可能导致索引功能异常，且难以排查。

**建议**：
- 为索引数据增加简单的校验和（如对关键字段计算哈希）
- 加载索引时校验数据结构完整性，损坏时自动回退到空索引并提示用户
- 在 `flushIndexNowAsync` 中增加写入后回读验证

**优先级**：低 | **实施难度**：中

#### 2.4.4 用户确认机制改进

**问题**：清空索引操作使用 `confirm()` 弹窗（content.js 第 1951、1963 行），这是阻塞式对话框，且无法自定义样式，与扩展的整体 UI 风格不一致。

**建议**：
- 使用自定义模态对话框替代 `confirm()`，与扩展的暗色主题一致
- 增加二次确认（如要求输入"确认清空"文字）
- 对不可逆操作增加操作日志

**优先级**：低 | **实施难度**：低

## 三、架构改进建议

### 3.1 状态管理统一

**问题**：当前状态分散在多处：
- content.js 的 `config` 变量（从 `chrome.storage.sync` 加载）
- content.js 的 `memberIndex` 变量（从 `chrome.storage.local` 加载）
- background.js 的 `indexCache` 变量（从 `chrome.storage.local` 加载）
- background.js 的 `indexingState` 变量（纯内存）
- popup.js 每次打开都重新从 storage 读取

没有统一的状态管理机制，状态同步依赖消息传递，容易出现不一致。

**建议**：
- 引入简单的状态管理模式：background.js 作为唯一状态权威来源
- content.js 和 popup.js 通过消息查询状态，而非各自维护副本
- 状态变更通过事件驱动，确保所有消费者同步更新

**优先级**：中 | **实施难度**：高

### 3.2 消息通信规范化

**问题**：消息类型使用字符串硬编码，请求/响应格式无约束，部分消息使用 `sendResponse` 回调，部分使用 `chrome.runtime.sendMessage` 广播，缺乏统一的通信协议。

**建议**：
- 定义消息类型常量枚举
- 为每种消息类型定义 TypeScript 风格的接口（即使不用 TypeScript，也以 JSDoc 形式记录）
- 统一通信模式：请求-响应用 `sendResponse`，事件通知用 `sendMessage` 广播
- 增加消息超时处理，避免回调永不触发

**优先级**：中 | **实施难度**：低

### 3.3 CSS 架构优化

**问题**：
- dark-mode.css 中大量使用 `!important` 覆盖原站样式，维护成本高且容易产生优先级冲突
- enhancer.css 中部分样式规则重复定义（如 `.snh48-panel-header` 在第 399 行和第 479 行各定义一次，属性不同）
- 暗黑模式的内联样式清理（JS 侧）与 CSS 选择器覆盖（CSS 侧）存在职责重叠

**建议**：
- 审查并合并重复的 CSS 规则
- 减少 `!important` 的使用：通过提高选择器特异性替代（如 `html.snh48-dark body .target-class`）
- 明确 JS 侧与 CSS 侧的职责边界：CSS 负责已知类名的覆盖，JS 仅处理动态内联样式

**优先级**：中 | **实施难度**：低

## 五、代码质量度量建议

为持续监控和改进代码质量，建议引入以下度量标准：

1. **文件行数上限**：单文件不超过 500 行（当前 content.js 2260 行、background.js 1180 行远超此限）
2. **函数行数上限**：单函数不超过 50 行（当前 `createFloatPanel` 约 350 行、`injectEmbeddedSearch` 约 400 行）
3. **圈复杂度**：单函数不超过 10（当前 `setupVideoShortcuts` 的 switch 分支约 15）
4. **重复代码率**：目标 < 3%（当前 DEFAULT_CONFIG 三处重复、颜色解析逻辑三处重复）
5. **`!important` 使用量**：目标 < 50 处（当前 dark-mode.css 中约 200+ 处）
6. **测试覆盖率**：目标核心逻辑 > 60%（当前仅 1 个测试文件，覆盖索引容量和空格键修复）
7. **ESLint 规则**：引入 `eslint:recommended` 配置，零 warning 零 error
8. **消息类型覆盖率**：所有消息类型 100% 有 JSDoc 文档

## 六、总结

本项目功能完整、实用性高，但代码层面存在典型的"快速迭代"技术债：单文件体量过大、重复代码、编码风格不统一、错误处理不充分。功能层面，最紧迫的改进是增量索引支持和 Service Worker 状态恢复，这两项直接影响核心功能的可靠性。

建议按 P0 → P1 → P2 的顺序逐步实施，每个优先级批次内可并行推进无依赖的改进项。代码拆分（1.1.1/1.1.2）应尽早进行，因为它是后续大部分改进的基础——模块化之后，其他优化才能在不增加耦合的前提下安全实施。
